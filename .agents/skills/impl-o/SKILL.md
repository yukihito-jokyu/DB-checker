# impl-o

承認済み設計を「レビュー・検証済みの実装」に変える司令塔として使う。調査、実装、レビューは必ず独立したサブエージェントへ分担し、メインエージェントは統合、承認取得、進行管理、記録を担当する。

## References

必要な場面だけ以下のreferenceを読む。frontendやGo/Wails固有の実装規約は、既存の専門skillを必要に応じて追加で読む。

| Reference | 読む場面 |
| --- | --- |
| [brief.md](references/brief.md) | 承認済み設計、受け入れ条件、触る範囲を実装前提へ圧縮するとき。 |
| [dispatch.md](references/dispatch.md) | 調査、レビュー、検証観点を分担した方が速いとき。 |
| [decide.md](references/decide.md) | 実装中に出た設計外判断を記録するとき。 |
| [review.md](references/review.md) | 差分、受け入れ条件、回帰リスク、テスト不足を確認するとき。 |
| [verify.md](references/verify.md) | 検証を実行し、結果または不能理由を残すとき。 |
| [note.md](references/note.md) | `.local/<issue>/impl.md` を作るとき。 |

## 実行計画

調査サブエージェントの結果を統合した後、実装前に必ず以下を出力し、ユーザー承認を得る。承認前に実装・編集を開始しない。

- 目的: 今回実装すること。
- 前提: 承認済み設計、受け入れ条件、触る範囲。
- 使用reference: 必要なreferenceだけ選び、選んだ理由を短く書く。
- サブエージェント: 調査、実装、レビューの担当を必ず別々にする。入力、期待成果物、禁止事項、統合方法を明記する。
- 記録: `.local/<issue>/impl.md` に残す内容。
- 検証: 実行予定のテスト、lint、buildに加え、ユーザーが行う手動検証の手順と各手順で確認すべき結果。
- 修正箇所の表示: ユーザーへ示すファイル・関数・指摘は、必ずクリック可能なローカルファイルリンクで示す。

## 手順

1. `references/brief.md` を読んだ調査サブエージェントに、実装対象、受け入れ条件、制約、未知点を調査させる。
2. メインエージェントが調査結果を統合し、実行計画を提示してユーザー承認を得る。
3. 独立した実装サブエージェントにだけ編集を依頼する。実装中の設計外判断は `references/decide.md` に従い、重要なものはユーザーへ戻す。
4. 独立したレビューサブエージェントに、`references/review.md` に従う差分レビューを依頼する。実装担当をレビュー担当にしない。
5. 初回レビューで指摘があれば、新しい実装サブエージェントに修正を依頼し、別のレビューサブエージェントで再レビューする。
6. 再レビュー（2ターン目）でも指摘があれば、修正を続けず、指摘と選択肢をユーザーへ差し戻す。指摘がない場合だけ検証へ進む。
7. `references/verify.md` に従って自動検証とユーザー向け手動検証を整理し、`references/note.md` に従って `.local/<issue>/impl.md` を作る。

## 完了条件

- 実装、自己レビュー、検証が完了している。
- 作業内容、判断、検証結果が `.local/<issue>/impl.md` にある。
- ユーザーが実施する操作、各操作で確認すべき結果、未実施の場合の理由が `.local/<issue>/impl.md` と完了報告にある。
- ユーザーへ修正箇所を示す場合、対象ファイルと行をローカルファイルリンクで示している。
- 調査、実装、レビューが独立したサブエージェントで実施され、レビュー回数と結果が記録されている。
