# Review

テスト通過とは別に、変更の妥当性を確認する。

## 観点

- 受け入れ条件を満たしているか
- 設計から逸脱していないか
- 不要な変更や過剰な抽象化がないか
- 既存動作を壊す可能性がないか
- エラー、空状態、境界値を扱っているか
- テストや手動確認に不足がないか
- 変更対象のGo関数・メソッドに、短い一行の直前コメントがあるか

## 出力

- 指摘
- 対応済み事項
- 残リスク
- 追加検証候補
- レビュー回数（初回／再レビュー）

## ルール

- findings firstで重大度順に整理する。
- ユーザーへ指摘または修正箇所を示す場合は、`[ファイル名](/絶対パス:行番号)` 形式のクリック可能なローカルファイルリンクを使う。パス文字列だけを示さない。
- 問題なしの場合も、残るテストギャップを明記する。
- 初回レビューの指摘は、新しい実装サブエージェントによる修正へ渡す。
- 再レビューで指摘がある場合は、メインエージェントがユーザーへ差し戻す。
